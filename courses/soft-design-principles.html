SOLID

SOLID is a set of design principles to encapsulate behavior:

  • Single-purpose responsibility
  • Open-closed
  • Liskov substitution
  • Interface segregation
  • Dependency inversion

What does that mean?

SOLID with Power Plugs

Single-purpose responsibility:

A module should have one, and only one, reason to change.

Open-closed:

A module should be open for extension but closed for modification.

Liskov substitution:

Derived modules must be usable through the base module interface, without the need for the user to know the difference.

Interface segregation:

Many client specific interfaces are better than one general purpose interface.

Dependency inversion:

Depend upon Abstractions. Do not depend upon concretions.

SOLID with Classes

SOLID principles are defined in terms of software modules, i.e. classes or sets of classes (packages, components).

They in fact also apply to entire programs with network interfaces (such as Web APIs).

Object-oriented programming is about managing dependencies, by selectively reinverting certain key dependencies in your architecture so that you can prevent rigidity, fragility, non-reusability.

Imperative programming allows only one type of dependency: hierarchies.

Thanks to polymorphism, object-oriented programming allows arbitrary dependencies among components.
